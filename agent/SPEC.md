# SPEC

Product spec for Truly Homely, derived from `agent/GOAL.md` and `PLAN.md`. This
expands the north star into concrete, implementable decisions for a single
menu-browsing sprint (plus an optional polish sprint). Keep it simple: this is a
browse-only menu site, no ordering, no cart, no CMS.

## 1. Architecture

- **Framework:** Next.js App Router + TypeScript, already scaffolded under
  `src/app/`. Tailwind v4 for styling, `motion` (framer-motion) for animation.
  Both are installed; no new dependencies are needed for the core sprint.
- **Rendering:** fully static (SSG). The menu is known at build time, so the
  index and every dish detail view are statically generated. The detail route
  uses `generateStaticParams()` over the dish slugs. No runtime data fetching,
  no API routes, no server state.
- **Images:** `next/image` for every photo so we get responsive `srcset`,
  lazy-loading below the fold, and intrinsic sizing (no layout shift, no
  squish). Above-the-fold grid images may use `priority` sparingly. Photos are
  static assets under `public/menu/<slug>/...` and are referenced by absolute
  paths already present in the data (e.g. `/menu/khaman-dhokla/1.jpg`).
- **Why this shape:** the dataset is tiny (19 dishes, 31 photos) and never
  changes at runtime. SSG + static images is the fastest, simplest thing that
  meets "fast and mobile-first" with zero infrastructure.

## 2. Data model (single source of truth)

The data is already extracted to `src/data/menu.json`, generated by
`scripts/build-menu.mjs` from the Notion export in `data/source/`. The script
runs at `predev` and `prebuild` (see `package.json`), so **adding or changing a
dish or photo requires zero code changes**: drop files in the source, the JSON
regenerates, and the static pages rebuild. The app must import the generated
JSON and never hardcode dish names, counts, categories, or photo paths.

Top-level shape:

```
{
  generatedFrom: string,
  count: number,              // 19
  categories: string[],       // ["Breakfast","Snack","Dessert","Dinner","Lunch","Meal"]
  items: MenuItem[]
}
```

`MenuItem`:

```
{
  slug: string,        // stable id and URL segment, e.g. "khaman-dhokla"
  name: string,        // display name, e.g. "Khaman Dhokla"
  price: string | null,// pre-formatted, e.g. "₹100.00"; usually null
  tags: string[],      // e.g. ["Navarathri Stall"]; often empty
  types: string[],     // categories this dish belongs to; MAY be empty
  images: string[]     // 1+ absolute public paths; images[0] is the cover
}
```

Derived rules the implementation must honor (driven by data, not assumptions):

- **Categories** come from the top-level `categories` array (or equivalently the
  union of all `types`). Do not hardcode the list; render whatever is in the
  data.
- **"All" is the default filter** and shows every one of the `count` items.
- **Dishes with empty `types`** (e.g. Gujarati Muthiya, Pav Bhaji, Momos) belong
  to no category. They appear under "All" and must NOT silently disappear; they
  simply do not match any specific category filter. There are several of these,
  so this is a real case, not an edge case.
- **Price** is optional. Only one dish currently has a price. When `price` is
  null, render the dish cleanly with no price and no empty/placeholder slot.
- **Description:** the data has no description field. Do not invent one. `tags`
  (e.g. "Navarathri Stall") are the only extra dish metadata and may be shown as
  small labels/badges; absence of tags is the common case and must look clean.
- **Cover image** is `images[0]`. The detail view shows all `images`. Every dish
  has at least one image.

## 3. Routes

- `/` - menu index: header/hero, category filter bar, and the dish grid.
- `/dish/[slug]` - per-dish detail view: large photo(s), name, price (if
  present), tags (if present), and a back affordance to the index. Statically
  generated for all 19 slugs. An unknown slug returns Next's 404.

The index is the home page; no separate per-category routes are required
(filtering is client-side on the index, see below). Keep the surface minimal.

## 4. Browsing UX

Warm and homely look and feel is a fixed constraint: cozy, appetizing, warm
tones, soft rounded cards, generous spacing, big food photos. Not stark-minimal,
not neon. The dataset is Indian/Gujarati home cooking; copy and palette should
feel like a home kitchen, not a corporate restaurant chain.

**Index layout**

- A short warm header/hero: the "Truly Homely" wordmark and a one-line tagline.
  Keep it compact so dishes are near the top, especially on mobile.
- A **category filter bar**: an "All" control plus one control per category from
  the data. Selecting a category filters the grid to dishes whose `types`
  include it; "All" shows everything. Exactly one filter is active at a time.
  The active control is visually distinct. Filtering is instant and client-side
  (no navigation, no reload).
- A **dish grid**: responsive, photo-forward cards. Mobile-first: 1 column on
  narrow phones, scaling up to 2-3+ columns on wider viewports. Each card shows
  the cover photo (consistent aspect ratio, object-fit cover, no stretch), the
  dish name, the price when present, and tags when present. Cards are links to
  the dish detail view. The whole card is the tap target.
- An empty state is effectively unreachable (every category has dishes), but if
  a filter ever yields zero matches the grid shows a deliberate "no dishes here"
  message rather than a blank area.

**Detail view (`/dish/[slug]`)**

- Large hero photo (`images[0]`), then the remaining photos if the dish has more
  than one (a simple stacked gallery or scroller; no heavy lightbox needed).
- Dish name as the page heading, price when present, tags when present.
- A clear back link/button returning to the index. Browser back also works.
- Photos use the same well-framed, non-squished treatment as the grid.

**Mobile-first and overflow:** the layout must read beautifully at 390px with no
horizontal overflow on any page. Tap targets are comfortable; the filter bar
wraps or scrolls horizontally without breaking layout.

## 5. Animation language

Tasteful, intentional motion via `motion`. Three coordinated moments:

- **Entrance:** dish cards fade/rise in on the index, gently staggered, once on
  load. Subtle (small translate + opacity), fast, never blocking interaction.
- **Hover/active:** cards lift slightly and/or the photo scales a touch on hover
  and show a clear active/pressed state; interactive controls (filters, links)
  have visible hover/focus states.
- **Filtering:** changing category animates the grid (items entering/leaving)
  smoothly rather than snapping, e.g. layout/presence transitions. Must not
  cause layout jank or horizontal shift.
- **Detail open:** navigating to a dish feels connected to its card (entrance
  transition on the detail hero; a shared-element style transition is a nice
  bonus, not required).

**Reduced motion:** when `prefers-reduced-motion: reduce` is set, all of the
above degrade to no/near-instant transitions. Content is fully visible and
usable with motion disabled. Motion never gates content rendering.

## 6. Performance budget

- Static HTML; minimal client JS. Only the interactive pieces (filter bar, card
  hover/entrance, detail gallery) ship as client components; the rest stays
  server-rendered.
- All photos go through `next/image` with appropriate `sizes` for the grid
  columns so phones do not download desktop-sized images. No raw `<img>` for
  menu photos.
- No layout shift from images (reserved aspect ratio). No blocking webfonts that
  cause invisible text; use system or `next/font` with a fallback.
- Target: index interactive quickly on a mid-range phone; no jank during
  entrance or filter animations.

## 7. Evaluability contract (stable DOM hooks)

The Playwright evaluator must assert structure without reading pixels or source.
Implement these stable hooks exactly:

Index page:

- A grid container with `data-testid="dish-grid"` and a
  `data-count` attribute equal to the number of dish cards currently rendered
  (updates as filters change).
- Each dish card carries `data-testid="dish-card"` and `data-slug="<slug>"`, and
  is a link to `/dish/<slug>`. The card exposes the dish name as visible text.
  Cards with a price expose it as visible text; cards with tags render them as
  visible text.
- A filter bar with `data-testid="category-filter"` containing one control per
  category plus All. Each control has `data-category="<name>"` (use
  `data-category="all"` for All) and an active-state hook
  `data-active="true|false"`. Exactly one has `data-active="true"` at a time.
- Selecting a category sets `data-count` on the grid to the number of dishes
  whose `types` include that category, and only those `data-slug` cards remain
  in the DOM (or visible). Selecting All restores `data-count` to the full
  count.

Detail page:

- A container with `data-testid="dish-detail"` and `data-slug="<slug>"`.
- The dish name as the page's main heading; price visible when present; every
  photo rendered as an `<img>` with `naturalWidth > 0`.
- A back affordance with `data-testid="back-to-menu"` that returns to `/`.

These hooks let the evaluator verify: total count matches `menu.json` count;
category counts match the data; filtering changes the rendered set; navigation
to/from detail works; no broken images.

## 8. Fallbacks and accessibility

- **No-JS / pre-hydration:** the index renders all dishes server-side so content
  and links work without JS; filtering is a JS enhancement. The detail pages are
  static and fully functional without JS.
- **Missing photo:** every dish currently has at least one image, but if an
  image path is ever absent the card must show a deliberate warm placeholder,
  not a broken-image icon.
- **Keyboard:** filter controls and cards/links are reachable and operable via
  keyboard with visible focus states; activating a card with Enter navigates to
  the detail view.
- **Semantics:** images have meaningful `alt` text (the dish name); the filter
  bar communicates its selected state to assistive tech (e.g. aria-pressed or
  equivalent); headings are well-ordered.
- **Reduced motion:** honored as in section 5.
